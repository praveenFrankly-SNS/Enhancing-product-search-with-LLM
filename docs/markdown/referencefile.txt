Exactly. This is actually how production semantic search systems are designed.

There are **two completely different pipelines**:

---

# Pipeline 1 — Data Preparation Pipeline (Offline)

This pipeline is executed when:

* Initial setup
* New products are added
* Products are updated
* Scheduled refresh (daily/hourly)

It is **not** executed for every customer search.

```text
PostgreSQL (WANDS + Enterprise Tables)
                │
                ▼
        Bronze Delta Tables
                │
                ▼
        Silver Delta Tables
      (Cleaning & Standardization)
                │
                ▼
      Gold Product Catalog
     (Feature Engineering)
                │
                ▼
      Embedding Generation
                │
                ▼
 Product Embedding Table
                │
                ▼
Databricks Vector Search Index
```

### Output

* Gold Product Catalog
* Product Embeddings
* Vector Search Index

Everything is now ready for searching.

---

# Pipeline 2 — Online Search Pipeline (Runtime)

This runs **every time a customer searches**.

```text
Customer Query
        │
        ▼
Query Understanding
(Intent Detection)
        │
        ▼
Query Rewrite / Expansion
(Optional)
        │
        ▼
Generate Query Embedding
        │
        ▼
Databricks Vector Search
        │
        ▼
Top-K Candidate Products
        │
        ▼
Metadata Filtering
(Category, Price, Brand, etc.)
        │
        ▼
Reranking
        │
        ▼
LLM Response Generation
(Optional)
        │
        ▼
Final Product Results
```

This pipeline usually takes only a few hundred milliseconds to a couple of seconds, depending on the LLM and reranking stage.

---

# Why are they separated?

Because embeddings are **expensive**.

Imagine 35,000 products.

If every customer search regenerated embeddings for all products:

```text
35,000 Products

↓

35,000 Embeddings

↓

Search
```

every search would take minutes and be prohibitively expensive.

Instead, we generate the **product embeddings once** and store them in the vector index.

At runtime, we only generate **one embedding**:

```text
Customer Query

↓

One Query Embedding

↓

Compare against existing Vector Index
```

This makes semantic search fast and scalable.

## One refinement I'd make

I would slightly rename the first pipeline to make its role clearer:

**Pipeline 1 – Product Indexing Pipeline (Offline)**

```text
PostgreSQL
    ↓
Bronze
    ↓
Silver
    ↓
Gold
    ↓
Embedding Generation
    ↓
Vector Search Index
```

This pipeline's purpose is to **build and maintain the searchable index**.

And the second becomes:

**Pipeline 2 – Semantic Search Pipeline (Online)**

```text
Customer Query
    ↓
Query Understanding
    ↓
Query Embedding
    ↓
Vector Search
    ↓
Hybrid Search
    ↓
Metadata Filtering
    ↓
Reranking
    ↓
LLM Response
    ↓
Results
```

This separation mirrors how enterprise search platforms (including Databricks AI Search) are typically organized: one pipeline prepares and indexes data, while the other serves low-latency search requests against that prepared index.
