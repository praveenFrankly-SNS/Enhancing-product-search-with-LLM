This is actually one of the most important architectural decisions we'll make because it determines whether the accelerator looks like a **real enterprise solution** or just a notebook demo.

After looking at the WANDS dataset structure you've shared, I **would not replace it with synthetic data**.

Let's first understand what WANDS already provides.

---

# WANDS is already a realistic dataset

The screenshots show three core tables.

### Product

Contains

* Product Name
* Product Class
* Category Hierarchy
* Product Description
* Product Features
* Rating Count
* Average Rating
* Review Count

This is already a fairly realistic product catalog.

---

### Query

Contains

Real customer search queries.

Example

```
salon chair

smart coffee table

chair and a half recliner

turquoise pillows
```

These are much better than generating random prompts using Faker.

---

### Label

Contains

```
Query

↓

Product

↓

Exact

Partial

Irrelevant
```

This is the hardest dataset to obtain manually.

---

# The problem isn't WANDS.

The problem is

**WANDS is static.**

It represents

```
One retailer

↓

One snapshot

↓

One benchmark
```

A production retailer is

```
Products added

↓

Products updated

↓

Prices change

↓

Descriptions improve

↓

Ratings change
```

---

# Should we generate synthetic products?

I would say

## ❌ No.

Reason

If we generate products like

```
Gaming Chair XYZ-984

Ultra Super Table

Comfort Sofa Deluxe
```

using Faker or GPT,

we lose

* realistic product descriptions
* realistic attributes
* realistic language
* realistic query relevance

Most importantly

our labels become meaningless.

---

# Should we extend WANDS?

## ✅ Yes.

This is what I would do.

Instead of replacing products,

we enrich them.

---

# Example

Current WANDS

| Product      | Description                    |
| ------------ | ------------------------------ |
| Office Chair | Mesh chair with lumbar support |

Enhance to

| Product      | Description                    | Price   | Brand         | Material | Color |
| ------------ | ------------------------------ | ------- | ------------- | -------- | ----- |
| Office Chair | Mesh chair with lumbar support | ₹12,999 | Herman Miller | Mesh     | Black |

Notice

The product itself stays untouched.

We're only enriching it.

---

# Same with Reviews

Current

```

Review Count
```

Enhancement

```

Top Negative Points
```

This adds richer semantic context without changing the benchmark itself.

# Should we generate reviews?

Yes.

This is one place where synthetic generation is useful.

Example

Instead of

```
Rating

4.8
```

Generate

```
Excellent lumbar support.

Comfortable for long office work.

Assembly takes about 20 minutes.
```

These reviews become additional searchable text.

Even Databricks mentions using product descriptions and user prompts as richer textual signals; adding review summaries extends that idea while keeping the original benchmark intact.

Price isn't part of semantic understanding,

but it's part of retrieval filtering.

# My recommendation

I would not create a synthetic product catalog.

I would create an **Enterprise Product Master** by enriching WANDS.

```
                 WANDS Product Catalog
                         │
        ┌────────────────┼────────────────┐
        │                │                │
   Preserve         Normalize        Enrich
        │                │                │
 Product Name      Features        Price
 Description       Categories      Brand
 Queries           Attributes      Review Summary
 Labels                           Specifications
        │
        ▼
 Enterprise Product Catalog
```

---

# One thing I would change from the Databricks accelerator

The accelerator treats WANDS as both the **catalog** and the **evaluation dataset**.

I think we should conceptually separate them.

```
                 WANDS
        ┌──────────┴──────────┐
        │                     │
Benchmark Dataset      Product Master
(Query + Labels)      (Enriched Catalog)
        │                     │
        └──────────┬──────────┘
                   ▼
        Product Search Pipeline
```

The benchmark portion (queries + relevance labels) remains untouched so you can fairly compare retrieval quality, while the product catalog is enriched with additional business attributes needed for a more realistic enterprise search experience. This gives you the best of both worlds: **realistic evaluation** and **production-oriented product data** without compromising the integrity of the original dataset.
